## Recent Pair Pressure Input Fixes

- Dive presentation preloads the Penguin fall animation in mascot `BeginPlay`, avoiding first-press synchronous loading. A per-dive recovery guard makes the get-up presentation idempotent. The grounded dive lock is a Fall Guys-style project approximation: hold the final dive pose for 0.30 seconds, then recover for 0.35 seconds, unlocking at 0.65 seconds total. Presentation reads the same component delay as gameplay so proxy timing cannot drift. A ledge handoff clears that delayed/already-playing dive recovery before playing the hanging loop, preventing a second get-up after dive-to-ledge capture. Authority resets any runaway positive ragdoll-root vertical velocity above 850 uu/s to a small downward value so a malformed constraint cannot accelerate upward indefinitely.
- Airborne locomotion zeroes the ground-only wobble before body-facing/air steering, preventing its changing yaw reference from twitching the mascot in flight. `AVNHShopperCharacter::CanJumpInternal` additionally requires a non-falling movement mode, preventing double jumps.
- Ragdoll startup uses the compact Penguin Physics Asset's two simulated bodies: `hips` and `chest`. The tightly limited hips-to-chest constraint preserves the body silhouette while allowing a real fall; head, hands, and toes remain rendered through animation but have no independent physics bodies, eliminating their jitter/spin. Core damping is deliberately low enough for normal gravity while retaining angular damping for a smooth fall. This shared path applies to both controlled Penguin knockdowns and the dummy throw.
- A dive can now target only an authored ledge/handle while airborne. A valid ledge publishes its hang state, then cancels the active dive and clears any pending dive-recovery montage; the dive gets a small additional forward launch plus a 1.12x montage that skips its inert opening frames.
- Tagged `PP.Grab.Pushable` blocks receive an all-direction proximity candidate pass and bypass only the camera/facing target gate; their existing range, mass, line-of-sight, and collision checks remain in force.
- The dedicated grab dummy throws through `UPPPhysicalStateComponent::RequestThrownRagdoll`: it clamps inherited jump vertical velocity to ±120 uu/s, uses 70% of the charged launch curve, and applies only a 75°/s facing-aware spin to the simulated hips root (clamped to 90°/s). This prevents an airborne throw from injecting unstable energy into both core bodies. It must not be thrown with `LaunchCharacter`, which only drives its character animation/movement path.
- Air-dive validation accepts an ascending jump even before the arm notification has reached authority, removing the jump-to-dive RPC ordering delay while still requiring a falling character and preserving the regular one-dive guard.
- Pair Pressure LMB release is intentionally idempotent: `HandlePairPressureGrabReleased` always relays to the grab component even if Enhanced Input has already cleared its local input-down flag, and the local keyboard/input poll supplies a mouse-up fallback for missed cancellation events. This prevents a released ledge hang from retaining its grab presentation.
- Charged Q throws for native grab targets now use `UPPGrabberComponent::ServerThrowHeldItemCharged`, rather than relying on the PlayerController's generic held-prop RPC. The component owns `CarriedGrabDummy` on authority, so this preserves the actual charge amount and executes the dummy's dedicated launch path.
- Real-player carry now matches the dedicated dummy's visible height by capturing the selected hips body's initial actor-relative height, targeting that offset from the dummy actor-root carry point, and ignoring world contacts only while the Physics Handle owns the incoming player. This prevents the floor from propping the simulated hips above the dummy while preserving the real-player physics/escape path. Normal release restores `CharacterMesh`; throw cleanup hands off to `UPPPhysicalStateComponent`, which restores full ragdoll collision before applying the shared dummy throw profile.
- A joining client's predicted grab broadcasts `Reaching` without locally writing the replicated `GrabState`, so the state can legitimately remain `None` while `ServerBeginGrab` is in flight. Q press/release must therefore treat `LMB held + None/Reaching` as a pending native grab. The ordered reliable component RPCs let the server resolve grab before charged throw under latency/packet loss; if acquisition is rejected, the throw request safely no-ops. A two-client PIE pass with 120 ms packet lag and 5 percent packet loss measured real-player authority/owner carry within about 1.3 units of the dummy target, delayed remote ragdoll entry as expected, smooth flight correction, and exact grounded recovery alignment.
- Falling locomotion uses full camera-relative forward-plus-A/D input, with the control multiplier applied only through `CharacterMovement.AirControl`; do not apply a second input multiplier or directly overwrite `Velocity`, as either makes air strafing imperceptible or causes visible twitching.
- Pair Pressure uses `JumpMaxHoldTime = 0` and `JumpMaxCount = 1`, so every jump is one fixed-height impulse. The native input path never calls `ACharacter::Jump` while falling; it records a held landing request and consumes it only after `OnMovementModeChanged` confirms the character is grounded. Releasing Space clears that request without calling `StopJumping`. This preserves instant jump-land-jump input without creating a midair `bPressedJump` saved-move hitch.
- Physical throw recovery reserves 0.20 seconds for continuous low-velocity grounded confirmation followed by the existing 0.28-second standing blend; both are scheduled inside the authored minimum lockout instead of being added after it. The disabled capsule relocates to the traced floor while the rendered throw-ragdoll pose is preserved, frozen-body collision is disabled, and the mesh/physics weight blends to standing. Course hazards no longer enter Chaos: authority selects Forward/Backward/Left/Right from the impact vector and replicates that direction plus synchronized fall/recovery timestamps. Every peer plays the same directional montage while the intact capsule stays in CharacterMovement, residual planar/upward velocity is cleared, and normal gravity lands an airborne hit. Grounded confirmation and the visual recovery remain inside the same 0.75-second response, eliminating the capsule-disabled hover/stuck failure by construction. Sampled obstacle point speed still gates eligibility independently of character facing; the 5 uu/s motion floor admits genuinely moving obstacles while a player running into stationary geometry cannot trigger a knockdown.
- `AVNHShopperCharacter` enforces `JumpMaxHoldTime = 0` and the baseline `JumpZVelocity = 650` both on its CDO and in `BeginPlay`, so stale inherited Blueprint defaults cannot produce variable jump heights. Physical ragdoll sources remain duration-bound: debug ragdoll is 1.25s, ordinary requests are 0.75–1.35s by daze, and dummy/player throws use the velocity-derived 0.75–2.20s window. Course-obstacle knockdowns are a separate fixed 0.75-second animation-driven path. While either path is down, additional impacts are ignored so an obstacle cannot repeatedly re-time the state. The compact hips/chest asset and its 4.0/4.5 angular damping remain exclusive to real throws/debug ragdolls. Dive recovery starts at 0.30s and unlocks at 0.65s after landing.

Course obstacle follow-up: runtime matching recognizes `Spinner_V2`, `SwingBall`, `SwingHammer`, `Turntable`, and the native `APPRotatingHammer` as course knockdown hazards. `Pusher_V1` reads `Speed`; Pusher V2 reads `Pusher1Speed` for `StaticMesh`, `Pusher2Speed` for `SM_Pusher_V1`, and `Pusher3Speed` for `StaticMesh1`; its stationary floor panel is explicitly excluded. Because these Blueprint timelines use unswept relative transforms, each authority-side impact sensor samples the registered moving components at 20 Hz and derives linear plus angular velocity at the exact contact point. A course knockdown is allowed only when that contacted obstacle point is moving by at least 5 uu/s; rotating/propeller hits gate on point-speed magnitude, while persistent pusher overlaps additionally require closing motion. Player movement into stationary geometry still cannot pass the gate. Runtime-spawned builder obstacles register through the server world's actor-spawn delegate. Drop V1 has no wind path: only hits on `SM_Propeller_V1` or `StaticMesh7` qualify. The response horizontally depenetrates the intact capsule by at most 18 units, ignores the triggering component until recovery, clears residual planar/upward CharacterMovement velocity, and never creates simulated skeletal bodies. `DT_MascotAnimations` exposes `ObstacleFallFront`, `ObstacleFallBack`, `ObstacleFallLeft`, and `ObstacleFallRight`; existing Penguin rows receive falls1/falls1/falls2/falls3 defaults, with falls1 intentionally serving as the forward fallback until a dedicated forward clip exists. Authored obstacle speed never changes travel or duration. The Lobby label `BP_Bridge_V2` is an instance of `BP_Bridge_V1`; both instances are native ledge targets that select the nearest real cooked-collision edge and align the stationary hanging pose/ledge-only hand IK to that point rather than an approximate actor-bound point. `BP_Floor_V1` collapses only the stepped movement-base tile after 0.5 seconds, including its collision. The local camera shortens its boom and disables boom collision while inside `SpinningTube`, Pusher V2, or Drop V1 bounds. The tube additionally uses collision-safe automatic crouch at full run speed on approach and restores the previous crouch settings on exit so the standing capsule can traverse the authored opening.

## Customizer Implementation Direction

Target behavior:
- Category rail selects an `EVNHCustomizationSlot`.
- Center grid displays the first page of items for that category.
- `LookItemSlot_1..20` should represent actual assets in the current category, not generic previous/next buttons.
- Clicking an item applies that exact mesh to the active customization and previews on the local pawn.
- Previous/next controls page through category items.
- Presets remain separate from item-grid options.
- Random randomizes the active customization from available Creative_Characters assets.
- READY marks pre-round ready and hides/locks the customizer when round starts.
- The character preview is rendered by `AVNHCustomizationPreviewActor` into a portrait render target and bound to `PreviewRenderImage`. Keep the render target aspect aligned with the right-side preview panel; a square render target stretches the character vertically in UMG.
- While the customizer is open, `VNHPlayerController` should suppress world interaction polling/focus traces and force the normal cursor so NPCs/interactables do not steal hover state through the UI.
- The designer widget currently stores preset buttons in `ItemGrid`, but `UVNHCharacterCustomizerWidget` moves them into a runtime bottom `PresetControlsBar_Runtime` and then normalizes `LookItemSlot_1..20` to a 5-column grid starting at row 0, column 0.
- Pagination arrows are hidden/disabled when a category has only one page.
- Preview rotation accepts right-click drag over `PreviewRenderImage` while preserving left-click drag as a fallback.
- Runtime preset edits are immediate: selecting a preset makes it active, and item changes auto-save to that active preset. Preset buttons should visibly show the active selection. `SAVE PRESET` and `LOAD PRESET` are intentionally hidden/removed from the current UX. `BLANK CANVAS` remains and clears all cosmetic meshes and face/hair while keeping the selected body mesh/body color.
- Preview rotation must rotate the character mesh rig only, not the camera/lights. `AVNHCustomizationPreviewActor` uses a fixed capture/light root plus a nested `CharacterRoot` for yaw.
- Preview animations should use the manually retargeted assets in `/Game/TNG/Characters/Animations/TryingOnClothesAnims/Male` and `/Game/TNG/Characters/Animations/TryingOnClothesAnims/Female`. The customizer exposes a Female Preview Anims checkbox under the preview panel to choose which set is preferred; if the selected set has no compatible clip, fall back to the other set/procedural try-on flourish.
